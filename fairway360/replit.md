# Fairway360

A premium marketing site + interactive demo + working multi-tenant SaaS for Fairway360, an AI operating system for golf courses and country clubs (powered by Scalaro). Full-stack: React/Vite frontend, Express 5 + PostgreSQL (Drizzle) backend, contract-driven via OpenAPI/Orval. Every business table carries `club_id`; Augusta Pines is tenant #1 / the demo club.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` â€” run the API server (port 5000)
- `pnpm run typecheck` â€” full typecheck across all packages
- `pnpm run build` â€” typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` â€” regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` â€” push DB schema changes (dev only)
- Required env: `DATABASE_URL` â€” Postgres connection string

## Architecture decisions

- Full-stack, multi-tenant: the marketing pages are static, but the three portals + demo form are wired to the real API (`@workspace/api-client-react` hooks â†’ Express â†’ Postgres). `club_id` is ALWAYS derived from the session (`req.auth`), NEVER from client input. Roles: supervisor / employee / member; middleware `requireAuth`, `requireRole(...)`, `requireStaff`.
- Contract-first: `lib/api-spec/openapi.yaml` is the source of truth; Orval generates `@workspace/api-zod` (Zod schemas) + `@workspace/api-client-react` (TanStack Query hooks). API DTOs mirror the frontend `portal-data.ts` interfaces. Regenerate with `pnpm --filter @workspace/api-spec run codegen`.
- The demo request form (`demo.tsx`) POSTs to `/api/leads/demo-request` (routed to the Augusta Pines demo tenant as a sales lead).
- F&B orders flow through the API + a per-club SSE stream (`GET /api/realtime/orders`, native `EventSource`): a member places an order â†’ it appears live on the employees/supervisor F&B board. The old `localStorage` orders store has been removed.
- v1 integrations are env-gated with graceful no-op fallbacks (so the demo runs without keys) and use direct REST (no new npm deps, per `minimumReleaseAge`): AI concierge (Anthropic, `lib/concierge.ts`), transactional email (Resend, `lib/email.ts`), SMS (Twilio, `lib/sms.ts`), payments (Stripe Checkout + webhook, `lib/stripe.ts`).
- Deploy is single-service (Replit autoscale): in production the Express server also serves the built frontend (`artifacts/fairway360/dist/public`) with an SPA fallback (`lib/static.ts`); in dev, Vite serves the client on 5173 and proxies `/api` â†’ 5000.
- Routing via `wouter` with `base` set from `import.meta.env.BASE_URL` so the app works under its artifact path prefix.
- CTAs use shadcn `Button asChild` wrapping wouter `Link` to avoid nested `<a><button>` markup (a11y).
- Solutions and Automations pages are data-driven from static arrays in-file; Automations uses an accordion (framer-motion `AnimatePresence`) with ARIA disclosure attributes.

## Design system â€” "The Members' Standard"

Heritage country-club editorial concept: warm/premium, not generic SaaS.

- **Type**: Fraunces serif for all headlines (loaded in `index.css`; a base-layer rule makes `h1â€“h4` serif via `--app-font-serif`). Inter for body/UI. Use `.font-display` to apply serif to non-heading elements (e.g. `CardTitle`, which is a `<div>` and is NOT caught by the h-tag rule).
- **Eyebrows**: small uppercase letterspaced labels above section headings via the `.eyebrow` utility. On dark forest sections use `text-accent`; on light ivory sections use `text-[hsl(38_55%_40%)]` (darker brass) for contrast.
- **Palette** (light tokens in `index.css`): background warm ivory `40 33% 96%`, foreground `155 30% 11%`, primary deep forest `155 52% 15%`, accent antique brass `40 58% 50%`, card `42 45% 99%`, border `40 20% 86%`. Radius `0.5rem`.
- **Dark sections**: deep forest `bg-[hsl(155_45%_8%)]` (or `155 44% 8%` for the dashboard panel) with `bg-white/[0.04]` cards and `border-white/10` â€” replaced the old cold `slate-950/900/800`.

### Portals â€” "Futuristic Clubhouse" (dark)

**All three** portals â€” Members (`/portal/members`), Supervisor (`/portal/supervisor`), and Employees (`/portal/employees`) â€” are **self-contained dark, mobile-first apps** sharing this one design language â€” there is no shared light `PortalShell` anymore (it was deleted). Luxury golf-tech SaaS feel (Tesla Ă— private club). Each page replicates the same primitives locally (its own `Glass` + `fade()` + dark header + `shiftDark`/`shiftDot` status maps) rather than importing from one another; `service-board.tsx` and the leads/bookings tables are dark to match. Gold icon-nav hero tiles: **Team** (supervisor) and **F&B Service** (employees). The `service-board.tsx` "New Order" dialog is intentionally left as a light modal overlay (a focused form over the dark blurred backdrop).
- **Base**: `bg-[#04130c]` with a fixed ambient backdrop (emerald radial glow + soft gold/emerald blur orbs at `-z-10`). White text throughout.
- **Glass panels**: the local `Glass` component â€” `rounded-2xl border border-white/10 bg-white/[0.04] backdrop-blur-xl` + deep shadow; it wraps a framer-motion `motion.div` with a shared `fade()` `whileInView` fade-up (staggered via an `i` index prop).
- **Gold accents** use the `accent` token (brass). Active nav/buttons: `bg-accent text-accent-foreground`; active icon-nav tiles get a gold glow `shadow-[0_0_28px_-8px_hsl(43_65%_55%/0.7)]`. The "Order Food" nav tile is always gold-highlighted (it's the headline action).
- **Navigation**: a persistent 6-tile icon nav (top) + a `fixed` bottom mobile nav (5 items, `pb-[env(safe-area-inset-bottom)]`); `main` has `pb-28` to clear it. Sections are **state-driven** (`section` state, not routes) and re-animate on switch via a `key={section}` motion wrapper.
- Because the page is dark, shadcn `Button variant="outline"` is overridden with explicit `border-white/20 bg-white/5 text-white` (the default outline assumes a light bg).
